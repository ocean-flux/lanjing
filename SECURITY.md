# Security Policy

## 报告安全漏洞

我们非常重视 LanJing 的安全性。如果你发现安全漏洞，请**不要**通过公开 Issue 报告。

### 报告流程

1. **使用 GitHub Security Advisories**（推荐）
   - 访问：https://github.com/ocean-flux/LanJing/security/advisories
   - 点击 "Report a vulnerability"
   - 提供详细的漏洞信息

2. **或通过邮件报告**
   - 发送到：（待补充安全邮箱）
   - 邮件主题：`[SECURITY] LanJing vulnerability report`

### 报告内容

请包含：
- 漏洞描述
- 影响范围
- 复现步骤
- 可能的修复建议（可选）
- 你的联系方式（如果希望保持联系）

### 请勿公开

在漏洞修复前，请**不要**：
- 在公开 Issue 中讨论漏洞细节
- 公开发布漏洞利用代码
- 在社交媒体或论坛公开漏洞
- 在未征得同意前向第三方披露

### 响应时间

- **初次响应**：7 天内
- **修复计划**：14 天内（取决于严重程度）
- **补丁发布**：30 天内（严重漏洞会加速）

## 安全边界

### 1. 规则执行安全

LanJing 是一个**规则驱动的媒体发现工具**，用户可以导入和编写规则来访问各种内容源。

#### 规则能力边界

**规则可以**：
- 发起 HTTP/HTTPS 请求
- 执行受限的 JavaScript 代码（在沙箱中）
- 使用 WebView 加载网页（受限）
- 读写本地缓存和 session
- 访问用户授权的 Cookie

**规则不能**：
- 执行任意系统命令
- 访问文件系统（除了规则包和缓存目录）
- 建立任意网络连接（仅限 HTTP/HTTPS）
- 修改系统配置
- 访问其他应用的数据

#### 网络访问风险

规则可以向**任何 URL** 发起 HTTP 请求，这意味着：
- **风险**: 恶意规则可能向攻击者控制的服务器发送数据
- **风险**: 规则可能请求大量数据消耗带宽
- **风险**: 规则可能访问内网地址（本地网络扫描）

**用户责任**：
- 只从可信来源导入规则
- 导入前审查规则内容
- 注意规则请求的权限范围

#### JavaScript 执行风险

规则可以执行 JavaScript 代码（通过 `rquickjs` 沙箱），但：
- 无法访问 Node.js API
- 无法访问文件系统
- 无法执行系统命令
- **注意**: 可能存在沙箱逃逸风险（欢迎报告）

#### WebView 风险

使用 `webviewDocument` profile 的规则会启动 WebView：
- **风险**: WebView 可以执行任意网页 JavaScript
- **风险**: 可能存在 XSS 或恶意脚本风险
- WebView 会话隔离，不共享主应用 Cookie
- WebView 无法访问文件系统（Tauri 限制）

### 2. 数据安全

#### 本地存储

LanJing 将数据存储在本地：
- **规则包**：`~/.LanJing/rules/`
- **缓存**：`~/.LanJing/cache/`
- **Cookie/Session**：`~/.LanJing/storage/`
- **日志**：`~/.LanJing/logs/`

**注意**：
- Cookie 和 session 以明文存储（使用操作系统文件权限保护）
- 日志可能包含 URL 和请求头（不包含完整响应体）
- 规则包可能包含敏感配置（如自定义 header）

#### 不收集的数据

LanJing **不会**：
- 发送遥测或使用统计
- 上传用户数据到云端
- 追踪用户行为
- 嵌入广告或分析 SDK

### 3. 第三方内容

LanJing 用于访问**第三方内容源**：
- **风险**: 内容源可能提供恶意内容
- **风险**: 内容源可能侵犯版权
- **风险**: 内容源可能不稳定或下线

**LanJing 不对第三方内容负责**，详见 [DISCLAIMER.md](DISCLAIMER.md)。

### 4. 依赖安全

我们使用 `cargo-deny` 审计 Rust 依赖：
```bash
cargo deny check
```

已知限制：
- 前端依赖未自动审计（欢迎贡献）
- 需要手动更新依赖版本

## 支持范围

### 支持的版本

| 版本 | 支持状态 |
| --- | --- |
| main 分支 | 持续支持 |
| 发布版本 | 最新版本支持 6 个月 |
| 旧版本 | 不支持 |

### 不在支持范围内

以下场景**不视为安全漏洞**：
- 用户导入恶意规则导致的问题
- 第三方内容源的问题
- 用户自行修改代码后的问题
- 在不支持的平台上运行
- DoS 攻击（本地应用）

## 安全最佳实践

### 用户建议

1. **只从可信来源导入规则**
   - 优先使用官方规则仓库
   - 导入前审查规则内容
   - 注意规则请求的权限

2. **定期更新**
   - 保持 LanJing 为最新版本
   - 更新会包含安全修复

3. **网络隔离**（可选）
   - 在虚拟机或容器中运行
   - 使用防火墙限制网络访问

4. **数据备份**
   - 定期备份规则和配置
   - Cookie/Session 丢失可重新登录

### 开发者建议

1. **审查规则**
   - 检查网络请求目标
   - 检查 JavaScript 代码
   - 检查权限声明

2. **报告可疑规则**
   - 在规则仓库报告
   - 提供证据和分析

3. **贡献安全检查**
   - 改进沙箱机制
   - 添加权限控制
   - 改进审计工具

## 已知限制

### 当前阶段（v0.1.0）

LanJing 处于早期开发阶段，以下功能**尚未完成**：
- 规则签名验证
- 规则权限细粒度控制
- 网络请求频率限制（基础实现中）
- 完整的沙箱审计
- Session/Cookie 加密存储

我们会在后续版本中逐步完善这些安全功能。
